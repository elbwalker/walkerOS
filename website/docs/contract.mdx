# Contract

A contract defines what your events should look like — which fields are required,
what types they have, and what values are allowed. It's a single source of truth
for event structure, shared across your entire flow configuration.

```json
{
  "version": 2,
  "contract": {
    "version": 2,
    "$tagging": 1,
    "globals": {
      "type": "object",
      "required": ["country"],
      "properties": {
        "country": { "type": "string", "description": "ISO country code" }
      }
    },
    "events": {
      "product": {
        "*": {
          "properties": {
            "data": { "required": ["id", "name"] }
          }
        },
        "add": {
          "properties": {
            "data": { "required": ["quantity"] }
          }
        }
      }
    }
  },
  "flows": { ... }
}
```

## Why contracts?

Without contracts, validation rules live inside each transformer config,
duplicated across flows and disconnected from your event design. Contracts solve
this:

- **Single source of truth** — Define event requirements once at the setup level
- **Shared across flows** — All flows inherit the same baseline rules
- **Self-documenting** — JSON Schema `description` and `examples` annotate your
  events for humans and tools
- **Versioned** — `$tagging` tracks contract versions alongside your events

## Contract versions

Contracts support two formats:

| Format | `version` field | Description |
|--------|----------------|-------------|
| **v2 (structured)** | `2` | Explicit sections mirroring `WalkerOS.Event` fields. Entity-action schemas live under `events`. |
| **Legacy (flat)** | absent | Flat entity-action map at the contract root. Still works, but v2 is recommended. |

Set `"version": 2` inside the contract to use structured sections:

<CodeSnippet
  code={`"contract": {
  "version": 2,
  "$tagging": 1,
  "globals": {
    "type": "object",
    "required": ["country"],
    "properties": {
      "country": { "type": "string", "description": "ISO country code" }
    }
  },
  "consent": {
    "type": "object",
    "required": ["analytics"],
    "properties": {
      "analytics": { "type": "boolean", "const": true }
    }
  },
  "events": {
    "product": {
      "add": {
        "properties": {
          "data": { "required": ["id", "quantity"] }
        }
      }
    }
  }
}`}
  language="json"
/>

## Cross-event sections

v2 contracts have top-level sections that describe properties shared across all events. Each section is a JSON Schema for the corresponding `WalkerOS.Event` field:

| Section | Event field | Purpose |
|---------|------------|---------|
| `globals` | `event.globals` | Cross-event key-value pairs (country, currency) |
| `context` | `event.context` | Timing/context properties |
| `custom` | `event.custom` | Custom properties |
| `user` | `event.user` | User identity and attributes |
| `consent` | `event.consent` | Consent state |

Top-level sections merge **additively** into per-event schemas. A per-event schema can tighten (add more requirements) but not loosen what the top-level section requires:

<CodeSnippet
  code={`"contract": {
  "version": 2,
  "consent": {
    "type": "object",
    "required": ["analytics"]
  },
  "events": {
    "product": {
      "add": {
        "properties": {
          "consent": {
            "required": ["analytics", "marketing"]
          }
        }
      }
    }
  }
}

// Resolved "product add":
// consent requires ["analytics"] (from top-level)
// + consent requires ["marketing"] (from per-event)
// = consent requires ["analytics", "marketing"]`}
  language="json"
/>

### `$globals` reference

Use `$globals` in the validator settings to validate `event.globals` on every event, even events without a matching contract entry:

<CodeSnippet
  code={`"transformers": {
  "validator": {
    "package": "@walkeros/transformer-validator",
    "config": {
      "settings": {
        "contract": "$contract",
        "globals": "$globals"
      }
    }
  }
}`}
  language="json"
/>

`$globals` is resolved at build time from the contract's `globals` section. The same pattern works for `$context`, `$custom`, `$user`, and `$consent`.

## Event schemas

Inside `events` (v2) or at the contract root (legacy), entity → action keyed
entries define JSON Schema objects describing a partial `WalkerOS.Event`:

<CodeSnippet
  code={`"contract": {
  "$tagging": 1,
  "product": {
    "*": {
      "description": "A product in the catalog",
      "properties": {
        "data": {
          "type": "object",
          "required": ["id", "name"],
          "properties": {
            "id": { "type": "string", "description": "Product SKU", "examples": ["SKU-123"] },
            "name": { "type": "string", "description": "Display name" }
          }
        }
      }
    },
    "add": {
      "description": "Product added to cart",
      "properties": {
        "data": {
          "type": "object",
          "required": ["quantity"],
          "properties": {
            "quantity": { "type": "integer", "minimum": 1, "description": "Items added" }
          }
        }
      }
    }
  }
}`}
  language="json"
/>

| Key | Description |
|-----|-------------|
| `$tagging` | Contract version number (optional, see [Versioning](#versioning)) |
| Entity key (e.g. `product`) | Groups actions for a specific entity |
| Action key (e.g. `add`) | JSON Schema for a specific entity-action pair |
| `*` | Wildcard — matches all entities or all actions |

## Wildcard inheritance

Contracts support four wildcard levels that merge **additively**:

| Level | Pattern | Matches |
|-------|---------|---------|
| 1 | `*` → `*` | All events (global rules) |
| 2 | `*` → `action` | A specific action across all entities |
| 3 | `entity` → `*` | All actions of a specific entity |
| 4 | `entity` → `action` | Exact match |

All matching levels combine. For `product add`, levels 1, 3, and 4 all apply:

<CodeSnippet
  code={`"contract": {
  "*": {
    "*": {
      "properties": {
        "consent": { "required": ["analytics"] }
      }
    }
  },
  "product": {
    "*": {
      "properties": {
        "data": { "required": ["id", "name"] }
      }
    },
    "add": {
      "properties": {
        "data": { "required": ["quantity"] }
      }
    }
  }
}

// Resolved "product add":
// consent.analytics required  (from *→*)
// data.id, data.name required (from product→*)
// data.quantity required      (from product→add)`}
  language="json"
/>

:::warning Contracts vs mapping wildcards
Contract wildcards use **additive merging** — all matching levels combine.
Mapping wildcards use **fallback matching** — the first match wins.

This difference is intentional. Contracts define cumulative requirements (entity-level
rules always apply to all actions), while mappings select a single transformation
target.

**Contract:** `product.*` rules AND `product.add` rules both apply to `product add`

**Mapping:** `product.add` matches first, so `product.*` is never checked
:::

## Merge algorithm

When multiple levels match, schemas merge with these rules:

| JSON Schema keyword | Merge behavior |
|---------------------|----------------|
| `required` | Union (deduplicated) — can only add requirements, never remove |
| `properties` | Deep merge — child wins on conflict for scalar values |
| Scalar keywords (`minimum`, `maximum`, `pattern`, etc.) | Child overrides parent |
| Annotations (`description`, `examples`) | Child replaces parent |

<details>
<summary>Merge example</summary>

Given a parent schema from `product.*`:

```json
{
  "properties": {
    "data": { "type": "object", "required": ["id", "name"] }
  }
}
```

And a child schema from `product.add`:

```json
{
  "properties": {
    "data": { "type": "object", "required": ["name", "quantity"] }
  }
}
```

The merged result for `product add`:

```json
{
  "properties": {
    "data": { "type": "object", "required": ["id", "name", "quantity"] }
  }
}
```

The `required` arrays are unioned and deduplicated: `["id", "name"]` + `["name", "quantity"]` = `["id", "name", "quantity"]`.

</details>

## Config vs settings level

Contracts cascade from config level (shared baseline) to settings level
(per-flow additions):

```
Flow.Config (version: 2)
  └── contract: { ... }          ← Shared across all flows
      └── Flow.Settings ("web-shop")
          └── contract: { ... }  ← Adds flow-specific rules
```

Settings-level contracts merge on top of the config-level contract using the same
additive algorithm. This lets you define common rules once and extend them per
flow:

<CodeSnippet
  code={`{
  "version": 2,
  "contract": {
    "product": {
      "*": {
        "properties": {
          "data": { "required": ["id", "name"] }
        }
      }
    }
  },
  "flows": {
    "web-shop": {
      "web": {},
      "contract": {
        "product": {
          "add": {
            "properties": {
              "data": { "required": ["price"] }
            }
          }
        }
      }
    }
  }
}

// "product add" in web-shop flow resolves:
// data.id, data.name required  (from setup product→*)
// data.price required          (from config product→add)`}
  language="json"
/>

## Versioning

The `$tagging` field tracks contract versions as an incrementing integer:

<CodeSnippet
  code={`"contract": {
  "$tagging": 3,
  "product": { ... }
}`}
  language="json"
/>

At build time, the bundler injects `$tagging` into the collector config so every
event gets `version.tagging: 3`. This lets you:

- Know which contract version validated an event
- Compare `event.version.tagging` against the current contract
- Track contract evolution over time

The `$` prefix signals metadata — it's not treated as an entity name during
resolution.

## Using contracts with the validator

The [validator transformer](/docs/transformers/validator) enforces contracts at
runtime. Use the `$contract` reference to automatically resolve and inject the
contract:

<CodeSnippet
  code={`{
  "version": 2,
  "contract": {
    "$tagging": 1,
    "product": {
      "*": {
        "properties": {
          "data": { "required": ["id", "name"] }
        }
      }
    }
  },
  "flows": {
    "default": {
      "web": {},
      "transformers": {
        "validator": {
          "package": "@walkeros/transformer-validator",
          "config": {
            "settings": {
              "format": true,
              "contract": "$contract"
            }
          }
        }
      }
    }
  }
}`}
  language="json"
/>

The `$contract` reference is resolved at build time by:

1. Collecting the setup-level contract (stripping `$`-prefixed metadata)
2. Merging the config-level contract on top
3. Resolving wildcard inheritance for each entity-action pair
4. Stripping annotations (`description`, `examples`) for the runtime schema
5. Passing the resolved rules to the validator

The validator itself doesn't change — `$contract` produces exactly the rule
structure it already expects.

## CLI validation

Validate contracts with the CLI or MCP server:

<CodeSnippet
  code={`# Validate a contract from a file
walkeros validate contract.json --type contract

# Validate inline
echo '{"product":{"add":{"properties":{}}}}' | walkeros validate --type contract

# Validate a full flow (includes contract validation)
walkeros validate flow.json`}
  language="bash"
/>

The contract validator checks:
- `$tagging` is a non-negative integer (if present)
- Entity and action keys are non-empty
- Each entry is a valid JSON Schema object
- v2: sections (`globals`, `context`, etc.) are valid JSON Schema objects

In the `web-shop` flow, `product add` merges these levels:

| Level | Source | Rules added |
|-------|--------|-------------|
| Top-level `globals` | Setup | `globals.country`, `globals.currency` required |
| Top-level `consent` | Setup | `consent.analytics` required |
| `product.*` | Setup events | `data.id`, `data.name` required |
| `product.add` | Setup events | `data.quantity` required, `consent.marketing` required |
| `product.add` | Config events | `data.price` required (number) |
