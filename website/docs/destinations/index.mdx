# Destinations

Destinations are the endpoints where your collected events are sent. WalkerOS
supports a wide variety of destinations for both web (browser) and server
environments.

## Destination Types

### [Web Destinations](./web)

Web destinations run in the browser and send events directly from the user's
device:

- **API** - Send to any custom HTTP endpoint
- **Google (Gtag)** - Google Analytics 4, Google Ads, GTM
- **Meta Pixel** - Facebook/Meta conversion tracking
- **Piwik PRO** - Privacy-focused analytics
- **Plausible** - Simple, privacy-friendly analytics

### [Server Destinations](./server)

Server destinations run in Node.js environments and are ideal for:

- Server-side tracking
- Sensitive data handling
- Batch processing
- Backend integrations

Available server destinations:

- **AWS** - Amazon Kinesis Firehose
- **GCP** - Google BigQuery
- **Meta Conversions API** - Server-side Facebook/Meta tracking

## Choosing Between Web and Server

### Use Web Destinations When:

- You need real-time user interaction tracking
- Client-side context is important (viewport, mouse position, etc.)
- You want simple implementation without server infrastructure
- The destination service provides a JavaScript SDK

### Use Server Destinations When:

- Handling sensitive data (PII, payment info)
- You need guaranteed delivery
- Batch processing is required
- You want to enrich data with server-side information
- Compliance requires server-side processing

## Common Concepts

### Event Mapping

All destinations support mapping to transform walkerOS events into the
destination's expected format:

```typescript
const mapping = {
  page: {
    view: {
      // Map walkerOS event to destination format
      event_name: 'page_view',
      page_location: 'context.location.href',
      page_title: 'data.title',
    },
  },
};
```

### Configuration

Every destination requires configuration:

```typescript
const config = {
  settings: {
    // Destination-specific settings
  },
  mapping: {
    // Event transformation rules
  },
};
```

### Error Handling

Both web and server destinations provide error handling:

```typescript
collector.on('error', (error) => {
  if (error.source === 'destination') {
    console.error(`Destination error:`, error);
  }
});
```

## Getting Started

1. Choose your destination type (web or server)
2. Install the destination package
3. Configure the destination with your credentials
4. Set up event mapping rules
5. Test the integration

## Best Practices

### Security

- Never expose sensitive credentials in web destinations
- Use environment variables for server destination credentials
- Implement proper authentication for custom API destinations

### Performance

- Load web destinations asynchronously when possible
- Batch events in server destinations
- Use appropriate transport methods (beacon for critical web events)

### Privacy

- Always respect user consent
- Implement destination removal when consent is revoked
- Document what data each destination receives

## Creating Custom Destinations

You can create custom destinations by implementing the destination interface:

```typescript
const customDestination = {
  type: 'custom',
  config: {},
  push(event, context) {
    // Your custom logic here
  },
};
```
